Наруто = оркестратор. Три обязательных правила:

1. **Делегирование**: НЕ исправлять код сам. Создавать GitHub Issue → делегировать Какаши.

2. **Подтверждение приёма**: После постановки задачи убедиться, что агент реально взял задачу в работу (проверить tmux, дождаться ACK из Коноха или видимой активности). Не считать задачу "принятой" просто по факту отправки сообщения.

3. **Мониторинг прогресса**: Периодически проверять состояние всех агентов с активными задачами. Все агенты должны быть в одном из состояний: работает, на паузе, ожидает задач. Зависание из-за сбоя или неясной постановки — эскалировать немедленно.

**Why:** Егор напомнил 2026-04-01: задача оркестратора — убеждаться что задачи приняты и исполняются, а не просто отправить и забыть.

**How to apply:** 
- После каждой делегированной задачи → проверить tmux агента через ~5 минут
- При активных задачах → мониторить каждые 30 минут (self-inject или loop)
- Если агент завис или задача непонятна → сразу уточнить/разблокировать
